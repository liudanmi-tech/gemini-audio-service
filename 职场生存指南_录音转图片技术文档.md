## 三、数据模型设计

### 3.1 核心数据表

#### 3.1.1 users (用户表)

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phone VARCHAR(11) UNIQUE,
    email VARCHAR(200) UNIQUE,
    password_hash VARCHAR(200),
    apple_user_id VARCHAR(100) UNIQUE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    last_login_at TIMESTAMP WITH TIME ZONE,
    is_active BOOLEAN DEFAULT TRUE
);
```

#### 3.1.2 sessions (会话/录音表)

```sql
CREATE TABLE sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title VARCHAR(255),
    start_time TIMESTAMP WITH TIME ZONE,
    end_time TIMESTAMP WITH TIME ZONE,
    duration INTEGER,  -- 秒
    status VARCHAR(50),  -- analyzing/archived/failed
    error_message TEXT,  -- 失败原因
    analysis_stage VARCHAR(100),  -- 当前分析阶段
    analysis_stage_detail JSONB,  -- 阶段详情
    emotion_score INTEGER,  -- 0-100
    speaker_count INTEGER,
    tags TEXT[],  -- 标签数组
    audio_url VARCHAR(500),  -- OSS音频URL
    audio_path VARCHAR(500),  -- 本地音频路径
    image_status VARCHAR(20) DEFAULT 'pending',  -- pending/generating/completed/failed
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_sessions_user_date ON sessions(user_id, DATE(created_at));
CREATE INDEX idx_sessions_status ON sessions(status);
```

#### 3.1.3 analysis_results (分析结果表)

```sql
CREATE TABLE analysis_results (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    dialogues JSONB NOT NULL,  -- 对话列表

    mood_score INTEGER,  -- 情绪分数

    conversation_summary TEXT,  -- "谁和谁对话"总结
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

#### 3.1.4 strategy_analysis (策略分析表)

```sql
CREATE TABLE strategy_analysis (
```

### 4.2 应用场景分类

录音转图片功能支持两种主要应用场景，针对不同的使用方式有不同的技术实现策略。

#### 4.2.0 场景对比

| 维度 | 场景1：多人对话 / 非自己说话 | 场景2：事后自述场景 |
|------|---------------------------|-------------------|
| **录音特征** | ① 多人实时对话，声纹可区分；或 ② 只有1人但声纹匹配的是别人（非用户自己） | 只有1人说话，且声纹匹配是用户自己 |
| **声纹识别** | 识别出多个说话人 或 单个说话人≠用户自己 | 单个说话人 = 用户自己（relationship="自己"） |
| **内容特征** | 真实对话交流，[我]和[对方]交替说话 | 用户事后口述，"今天我和张经理讨论了..." |
| **人物识别** | 基于声纹自动识别说话人 | 基于文本内容提取提到的人物名称 |
| **档案匹配** | speaker_mapping 自动映射 | 文本分析 + name字段模糊匹配档案 |
| **参考图策略** | 左侧=用户档案，右侧=对方档案 | 根据场景涉及的人物动态拉取对应档案图 |
| **典型场景** | 工作会议、汇报、谈判 | 日记式记录、复盘总结、故事讲述 |

> **检测判断规则**（优先级顺序）:
> 1. `speaker_count > 1` → **场景1**
> 2. `speaker_count == 1` 且 `该说话人声纹 ≠ 用户自己（relationship="自己"）` → **场景1**
> 3. `speaker_count == 1` 且 `该说话人声纹 = 用户自己` → **场景2**

#### 4.2.1 场景1：多人对话 / 非自己说话（原有逻辑不变）

**特征**:
- 录音中有多个人的声音；**或**录音只有1人但声纹匹配的是别人（非用户自己）
- Gemini可以识别出不同的说话人（Speaker_0, Speaker_1等）
- 通过声纹识别可以映射到具体的档案

**人物识别流程**:
```
1. Gemini音频分析 → 识别多个说话人
2. 声纹识别 → speaker_mapping
   例: {"Speaker_0": "profile_abc", "Speaker_1": "profile_xyz"}
3. 加载档案照片 → reference_images
4. 生成图片时使用档案照片作为参考
```

**参考图加载逻辑** (代码位置: `main.py` - `_get_profile_reference_images`):
```python
async def _get_profile_reference_images(session_id, user_id, db):
    """
    场景1：从speaker_mapping获取档案照片

    策略：
    1. 从 speaker_mapping 获取映射关系
    2. left_profile (用户): relationship="自己"的档案
    3. right_profile (对方): 映射到的其他档案
    4. 返回 [(left_bytes, mime), (right_bytes, mime)]
    """
```

#### 4.2.2 场景2：事后自述场景（故事讲述）⭐ 新增

**特征**:
- 录音中只有用户一个人的声音，且声纹识别确认该说话人是用户自己（`relationship="自己"`）
- 用户在事后口述今天发生的事情："今天我和张经理讨论了预算..."
- 整个 transcript 只有一个说话人，且为用户本人
- 需要从文本内容中提取用户提到的其他人物名称，拉取对应档案图作为参考

**进入场景2的判定条件**（在 `generate_scene_images` 入口处判断）:
```
speaker_count == 1
AND 该 speaker 在 speaker_mapping 中对应的 profile.relationship == "自己" / "Self"
```

**人物识别与参考图流程**:
```
1. Gemini音频分析 → 识别出1个说话人，speaker_mapping 验证为用户自己
2. 文本分析（Gemini） → 从口述文本提取提到的人物名称
   例: "今天我和张经理、李总监开会..." → ["张经理", "李总监"]
3. 档案匹配 → 根据 profiles.name 模糊匹配
   例: "张经理" → profiles.name ILIKE '%张经理%'
4. 直接拉取档案图 → 按场景描述判断涉及哪些人，加载对应档案照片作为参考图
   （有档案照片：直接用于多模态生成；无档案照片：智能降级，见下）
```

**技术方案要求**（与场景1保持一致）:
- ✅ **多模态生成**: 风格参考图 + 档案照片 + 文本描述，生成高一致性图片
- ✅ **智能降级**: 无档案照片时，用 Gemini 根据用户口述对该人物的描述推断其形象特征，构建 `consistency_header` 注入每张图片 prompt，保障多图人物一致性

```python
async def _extract_mentioned_people_and_match_profiles(
    transcript: list,
    user_id: str,
    gemini_flash_model: str,
    db: AsyncSession
) -> dict:
    """
    场景2专用：从用户自述中提取提到的人物并匹配档案

    Args:
        transcript: 对话转录列表（只有用户一个人说话）
        user_id: 用户ID
        gemini_flash_model: Gemini模型名
        db: 数据库会话

    Returns:
        {
          "张经理": {
            "profile_id": "uuid-123",
            "photo_bytes": b"...",
            "mime_type": "image/png",
            "relationship": "领导"
          },
          "李总监": {
            "profile_id": "uuid-456",
            "photo_bytes": b"...",
            "mime_type": "image/png",
            "relationship": "同事"
          }
        }
    """
    # Step 1: 提取用户讲述的完整文本
    user_narration = "\n".join([t.get("text", "") for t in transcript])

    # Step 2: 调用Gemini提取提到的人物名称
    prompt = f"""
分析以下用户的讲述内容，提取所有提到的人物名称。

讲述内容：
{user_narration}

规则：
- 提取所有具体的人名、称呼（如：张经理、李总监、小王、王老师）
- 排除泛指（如：大家、同事们、他们）
- 只返回JSON数组，格式：["张经理", "李总监", "小王"]
"""

    try:
        model = genai.GenerativeModel(gemini_flash_model)
        response = await asyncio.to_thread(model.generate_content, prompt)
        text = response.text.strip()

        # 解析JSON
        json_match = re.search(r'\[.*?\]', text, re.DOTALL)
        if json_match:
            mentioned_people = json.loads(json_match.group())
        else:
            mentioned_people = []

        logger.info(f"[场景2] 提取到的人物: {mentioned_people}")

        if not mentioned_people:
            return {}

        # Step 3: 从档案表匹配
        people_profiles = {}
        for person_name in mentioned_people:
            # 模糊匹配档案name字段
            query = await db.execute(
                select(Profile).where(
                    Profile.user_id == uuid.UUID(user_id),
                    Profile.name.ilike(f"%{person_name}%")
                ).limit(1)
            )
            profile = query.scalar_one_or_none()

            if profile:
                # 加载档案照片
                photo_data = _fetch_profile_image_from_oss(user_id, str(profile.id))
                if photo_data:
                    people_profiles[person_name] = {
                        "profile_id": str(profile.id),
                        "photo_bytes": photo_data[0],
                        "mime_type": photo_data[1],
                        "relationship": profile.relationship_type
                    }
                    logger.info(f"[场景2] 匹配成功: {person_name} -> {profile.name} ({profile.relationship_type})")
                else:
                    logger.warning(f"[场景2] 档案照片未找到: {person_name}")
            else:
                logger.warning(f"[场景2] 档案未匹配: {person_name}")

        return people_profiles

    except Exception as e:
        logger.error(f"[场景2] 提取人物失败: {e}", exc_info=True)
        return {}
```

```python
async def _match_scene_to_people(
    scene_description: str,
    people_profiles: dict,
    gemini_flash_model: str
) -> list:
    """
    场景2专用：判断某个场景涉及哪些人物

    Args:
        scene_description: 场景描述，如"用户正在向张经理汇报"
        people_profiles: 提到的人物档案字典
        gemini_flash_model: Gemini模型名

    Returns:
        涉及的人物名称列表，如 ["张经理"]
    """
    if not people_profiles:
        return []

    people_list = list(people_profiles.keys())

    prompt = f"""
场景描述：{scene_description}

可能涉及的人物列表：{people_list}

请判断这个场景中涉及哪些人物（可能是0个、1个或多个）。
只返回JSON数组，格式：["张经理"] 或 [] 或 ["张经理", "李总监"]
"""

    try:
        model = genai.GenerativeModel(gemini_flash_model)
        response = await asyncio.to_thread(model.generate_content, prompt)
        text = response.text.strip()

        json_match = re.search(r'\[.*?\]', text, re.DOTALL)
        if json_match:
            involved_people = json.loads(json_match.group())
            # 过滤：只保留在people_list中的
            return [p for p in involved_people if p in people_list]
        else:
            return []
    except Exception as e:
        logger.warning(f"[场景2] 匹配场景人物失败: {e}")
        return []
```

```python
            return None

    # 并行生成所有图片
    results = await asyncio.gather(
        *[gen_one(i, scene) for i, scene in enumerate(scenes)],
        return_exceptions=True
    )

    # 组装结果
    scene_images = []
    for i, (scene, img) in enumerate(zip(scenes, results)):
        if isinstance(img, Exception) or img is None:
            scene_images.append({
                "index": 1000 + i,
                "scene_description": scene,
                "image_url": None,
                "image_base64": None
            })
        else:
            is_url = img.startswith("http")
            scene_images.append({
                "index": 1000 + i,
                "scene_description": scene,
                "image_url": img if is_url else None,
                "image_base64": img if not is_url else None
            })
```

**性能数据**:
- 单张图生成时间: 5-15秒 (取决于prompt复杂度)
- 并行生成5张图: 15-20秒 (vs 串行75秒，提升75%)
- 超时保护: 单张120秒，避免无限等待

## 六、核心算法与技术要点

### 6.1 人物一致性算法

#### 6.1.1 有档案照片方案

**流程**:
```
1. 从 analysis_results.speaker_mapping 获取映射关系
   例: {"Speaker_0": "profile_uuid_123", "Speaker_1": "profile_uuid_456"}

2. 查询 profiles 表获取照片URL
   - left_profile (用户): relationship="自己"
   - right_profile (对方): 本次会话关联的其他档案

3. 从 OSS 下载照片字节数据
   路径: images/{user_id}/profile_{profile_id}/0.png

4. 构建 reference_images 列表
   [(left_bytes, "image/png"), (right_bytes, "image/png")]

5. 传入 generate_image_from_prompt() 作为参考图
```

**优势**:
- 精确的人物面部特征
- 高度一致性 (基于实际照片)
- 适合多场景、多对话的持续使用

#### 6.1.2 无档案照片方案

**流程**:
```
1. 收集对话样本
   - user_lines: [我]说的话 (前20句)
   - other_lines: [对方]说的话 (前20句)

2. 调用 Gemini 分析人物特征
   Prompt: "根据对话内容推断两位说话者的形象特征..."

3. 提取特征
   - 性别: 男/女
   - 年龄段: 20-30岁 / 40-50岁
   - 角色: 职场员工 / 领导
   - 外貌: "穿商务休闲装的年轻男性" / "穿正式西装的中年男性领导"

4. 构建一致性描述
   consistency_header = "左侧为穿商务休闲装的年轻男性（用户），
                        右侧为穿正式西装的中年男性领导（对方）。"

5. 将 consistency_header 注入每张图片 prompt 开头
   full_prompt = consistency_header + scene_description
```

**分析Prompt示例**:
```
请根据以下对话内容，推断两位说话者的形象特征。

【用户（我）说的话】：
领导，关于这个季度的预算，我想和您汇报一下...
目前我们部门的支出比计划多了15%...
是的，主要是因为新项目的启动费用超出预期...

【对方说的话】：

什么？怎么会多这么多？你是怎么管理的？
你要拿出一个整改方案来，下周一交给我。

请综合以下信息判断：称呼方式、说话语气、词汇习惯、提及的角色关系等。
只返回 JSON，格式如下（不要解释）：
{
  "user": {
    "gender": "男或女",
    "age_range": "如20-30岁",
    "role": "如职场员工",
    "appearance": "一句话外貌，如穿商务休闲装的年轻男性"
  },
  "other": {
    "gender": "男或女",
    "age_range": "如40-50岁",
    "role": "如领导/上司",
    "appearance": "一句话外貌，如穿正式西装的中年男性领导"

```

**优势**:
- 无需用户上传照片
- 自动化推断，降低使用门槛
- 适合首次使用或隐私敏感场景

**局限性**:
- 一致性低于照片方案 (文本描述 vs 视觉参考)

### 6.2 场景理解与提取

#### 6.2.1 场景提取原则

1. **画面感优先**: 选择有明确动作、表情、互动的场景
2. **情节完整**: 每个场景是一个完整的情节片段
3. **避免重复**: 不提取相似的场景
4. **数量控制**: 1-5个场景 (对话越长可提取越多)

#### 6.2.2 角色标注策略

**问题**: Gemini 音频分析返回的是 `Speaker_0`, `Speaker_1`，但图片生成需要明确"用户"和"对方"

**解决方案**:
```python
# 构建带角色标签的对话文本
lines = []
for t in transcript:
    speaker = t.get("speaker", "")
    text = t.get("text", "")
    is_me = t.get("is_me", False)  # 关键标志
    if text:
        label = "[我]" if is_me else "[对方]"
        lines.append(f"{label} {text}")

transcript_str = "\n".join(lines)

# 场景分析时明确要求使用"用户"和"对方"
scene_prompt = f"""
对话角色说明：
- [我] = 用户，画面中固定在左侧
- [对方] = 对话另一方，画面中固定在右侧

对话内容：
{transcript_str}

规则：
- 每个场景必须明确说明【谁】在做【什么动作】
- 使用"用户"和"对方"指代（不要用Speaker_0/1）
- 必须符合对话实际逻辑
"""
```

```python
    if error_code == 429:
        # 提取重试延迟
        retry_delay = 15  # 默认15秒
        delay_match = re.search(r'retry in ([\d.]+)s', error_message)
        if delay_match:
            retry_delay = max(15, int(float(delay_match.group(1))) + 2)

        logger.warning(f"配额超限(429)，等待{retry_delay}秒后重试...")

        # 检查是否是免费层配额为0
        if 'limit: 0' in error_message:
            logger.error("❌ 检测到免费层配额限制")
            logger.error("💡 建议检查:")
            logger.error("   1. 确认 API Key 是否关联到付费项目")
            logger.error("   2. 在 Google Cloud Console 检查配额设置")

        if attempt < max_retries - 1:
            time.sleep(retry_delay)
            continue
```

**客户端处理**:
```swift
let (asyncBytes, response) = try await session.bytes(for: request)
for try await line in asyncBytes.lines {
    if line.hasPrefix("data: ") {
        let content = String(line.dropFirst(6))
        if content == "[DONE]" {
            break
        } else if content.hasPrefix("[ERROR]") {
            throw NetworkError.serverError(content)
        } else {
            // 替换转义的换行符
            let text = content.replacingOccurrences(of: "\\n", with: "\n")
            yield text
        }
    }
}
```

### B. 常见问题 (FAQ)

**Q1: 图片生成需要多长时间？**
A: 单张图片 5-15秒，5张图片并行生成总计 15-20秒。

**Q2: 支持哪些音频格式？**
A: m4a, mp3, wav 等常见格式，最大 100MB。

**Q3: 场景图片一定会生成成功吗？**
A: 不一定。单张图片可能因超时、配额限制等失败，但不影响其他图片和整体分析。

**Q4: 如何提高人物一致性？**
A: 建议上传档案照片，系统会自动使用照片作为参考生成图片。

**Q5: 可以更换图片风格吗？**
A: 可以，在调用策略分析接口时传入 `style_key` 参数即可。

**Q6: 为什么有时候图片URL是null？**
A: 该场景图片生成失败（超时或错误），客户端应跳过展示。

**Q7: 读写分离是如何实现的？**
A: iOS客户端根据 `useBeijingRead` 配置，读请求走北京服务器，写请求走新加坡服务器。

**Q8: 如何查看服务日志？**
A: 新加坡服务器: `tail -f ~/gemini-audio-service.log`
北京服务器: `tail -f ~/gemini-audio-service-read.log`
